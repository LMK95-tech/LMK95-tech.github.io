01Analyse & Contexte

Avant l'intervention, les collaborateurs de SISR-Corp stockaient leurs documents localement sur leurs postes, sans aucune centralisation. Les quelques partages réseau existants accordaient le droit « Tout le monde — Contrôle total » sans distinction de service, rendant les dossiers Direction, RH et Technique accessibles à tous.

 5 risques critiques identifiés
Fuite de données
Un technicien peut lire les docs RH et Direction
Perte irrémédiable
Aucune sauvegarde centralisée — perte poste = perte data
Saturation incontrôlée
Sans quota, un user monopolise tout l'espace partagé
Risque ransomware
Exécutables .exe copiés librement — vecteur malware
 Solution : gouvernance centralisée WS2022
Permissions NTFS moindre privilège — chaque groupe AD accède uniquement à son dossier
ABE (Access-Based Enumeration) — dossiers non autorisés totalement invisibles
FSRM — quota 2 Go bloquant + filtres audio/vidéo/exécutables
02Arborescence & Permissions NTFS

L'arborescence est créée sur le volume E:\Partages\SISR-Corp avec un sous-dossier par département. L'héritage NTFS est explicitement désactivé pour chaque dossier afin d'isoler totalement les permissions entre services.

PowerShellCréation arborescence et permissions NTFS
# Création de l'arborescence des partages
PS C:\> New-Item -Path "E:\Partages\SISR-Corp\Direction" -ItemType Directory
PS C:\> New-Item -Path "E:\Partages\SISR-Corp\RH" -ItemType Directory
PS C:\> New-Item -Path "E:\Partages\SISR-Corp\Technique" -ItemType Directory
PS C:\> New-Item -Path "E:\Partages\SISR-Corp\Public" -ItemType Directory
# Désactivation de l'héritage et application des ACL NTFS (dossier Direction)
PS C:\> $acl = Get-Acl "E:\Partages\SISR-Corp\Direction"
PS C:\> $acl.SetAccessRuleProtection($true, $false) # Désactiver l'héritage
PS C:\> $rule = New-Object System.Security.AccessControl.FileSystemAccessRule(
"SISR-CORP\GS_Direction","Modify","ContainerInherit,ObjectInherit","None","Allow")
PS C:\> $acl.AddAccessRule($rule)
PS C:\> Set-Acl "E:\Partages\SISR-Corp\Direction" $acl

03ABE — Access-Based Enumeration

L'ABE (Access-Based Enumeration) est activée sur le partage réseau depuis Gestionnaire de serveur → Services de fichiers → Partages → Propriétés → Paramètres. Un utilisateur appartenant à GS_RH ne verra dans l'explorateur Windows que le dossier RH — les dossiers Direction et Technique sont totalement invisibles, même si l'utilisateur connaît leur chemin exact.

Exemple — Vue d'un utilisateur GS_RH dans \\SERVEUR-CORP\PartageDonnees
\\SERVEUR-CORP\PartageDonnees
RH← visible (droit accordé)
Public← visible (lecture publique)
Direction← invisible (ABE masque ce dossier)
Technique← invisible (ABE masque ce dossier)
L'ABE est une couche de sécurité par l'obscurité complémentaire aux ACL NTFS. Defense in depth.
04FSRM — Quotas & Filtres de fichiers
4.1 — Quota 2 Go par utilisateur (type Fort)

Depuis la console FSRM (File Server Resource Manager), un modèle de quota « Limite de 2 Go » de type Fort (bloquant) est créé et appliqué à D:\Donnees\SISR-Corp\Users. Un seuil d'alerte à 75% (1,5 Go) déclenche automatiquement un e-mail d'avertissement.

PowerShellVérification des quotas FSRM en temps réel
# Afficher tous les quotas actifs
PS C:\> Get-FsrmQuota | Select-Object Path, Size, Usage, SoftLimit
# Résultat attendu :

# 2 Go limite · 1,5 Go utilisé (75%) · Quota fort actif

06Conclusion & Bilan de gouvernance
 Données sécurisées — Gouvernance fichiers opérationnelle

La mise en œuvre du serveur de fichiers centralisé sur Windows Server 2022 pour SISR-Corp transforme radicalement la gouvernance des données : d'une anarchie incontrôlée à une infrastructure sécurisée, auditée et résiliente. Les permissions NTFS au moindre privilège, l'ABE et le FSRM constituent une défense en profondeur sur la couche données.
